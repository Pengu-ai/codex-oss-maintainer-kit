# Codex for OSS Application Playbook

This playbook helps maintainers prepare an evidence-bound application for maintainer support programs.

## Eligibility Checklist

- You are a primary maintainer, core maintainer, or repository admin.
- The repository is public and open source.
- The project has active maintenance evidence.
- The project has meaningful usage, adoption, or ecosystem importance.
- You can explain how Codex or API credits would reduce real maintainer workload.
- Claims can be backed by public links.

## Evidence To Collect

- Repository URL and license.
- Maintainer role and permission evidence.
- Stars, forks, dependents, downloads, citations, or known users.
- Recent releases and changelog entries.
- Representative issues and pull requests.
- Security or reliability work.
- Contributor onboarding and review burden.

## Strong Application Angle

Use this structure:

1. What the project does.
2. Who depends on it or why the ecosystem needs it.
3. What maintainer workload exists today.
4. Which Codex workflows would reduce that burden.
5. How you will keep outputs human-reviewed.

## Avoid

- Artificial star exchanges.
- Inflated adoption claims.
- Vague "AI will help coding" explanations.
- Claiming maintainer status without write/admin authority.
- Submitting private or confidential repository details.

## 30-Day Launch Milestones

- Week 1: publish v0.1.0, README, license, examples, and issue templates.
- Week 2: dogfood on real repos and publish packet examples.
- Week 3: invite maintainer feedback and merge first external contribution.
- Week 4: cut v0.2.0 with GitHub API enrichment or package registry evidence.
